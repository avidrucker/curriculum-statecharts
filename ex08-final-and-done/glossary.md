# Glossary — ex08: Final States, `done.state` Events, and Chart Exit

Terms introduced (or first deeply covered) in ex08. Cross-cutting vocabulary lives in the top-level [`glossary.md`](../glossary.md); ex01–ex07 covered states, transitions, parallels, internal/external types. This file adds what ex08 newly puts on the table.

Each entry has a 0–9 criticality rating (see the top-level glossary for the scale).

---

### `final` (element)  *(criticality: 9)*

The element from `com.fulcrologic.statecharts.elements` declaring a **terminal state**:

```clojure
(final {:id :validated})
```

Returns `{:id :validated, :node-type :final}`[^p1]. Distinguishing features:

- **No children allowed.** The library validates this at construction[^p6]; trying to declare a transition (or any child element) inside a final throws `Illegal children of :final ...`.
- **Auto-fires `done.state.<parent>`** on entry. The parent compound state's transitions can catch this event for cleanup or follow-on behavior.
- **Top-level finals exit the chart** when entered (Step 4 below).

Finals can be declared anywhere a state can: as a child of a compound state, as a child of a region, or at the chart root. The semantics differ by position (per-region completion vs chart exit).

### `done.state.<id>` (auto-event)  *(criticality: 9)*

An internal event synthesized by the runtime when a `final` element is entered. The `<id>` portion is the *final's parent*'s ID. For:

```clojure
(state {:id :container}
  (state {:id :working})
  (final {:id :done}))
```

Entering `:done` fires `:done.state.container`. Note: the event's name uses the *parent*'s id, not the final's own id.

The event shape[^p7]:

```clojure
{:type :internal
 :name :done.state.<parent-id>
 :sendid <final-id>
 :data {}
 :com.fulcrologic.statecharts/event-name :done.state.<parent-id>}
```

The `:type :internal` distinguishes it from external user events; the `:sendid` records *which* final triggered (useful when a compound has multiple finals).

A transition catches the event the same way it'd catch any event:

```clojure
(transition {:event :done.state.container :target :next-state})
```

Listening for the auto-event is how a chart reacts to "a region completed."

### parallel completion semantics  *(criticality: 8)*

For a parallel state with multiple regions, `done.state.<parallel-id>` fires only when **every** region has reached a final[^p5]. Partial completion fires the per-region `done.state.<region-id>` events, but not the parallel's overall one.

This makes the "wait for all parallel tasks to complete" pattern declarative: you don't track completion in the data model or check after every event. The runtime computes "all regions final" automatically and emits the broadcast event when (and only when) all are done.

For the exercise's chart: both `:validation` and `:enrichment` must reach their finals before `done.state.processing` fires. Once it fires, the transition on `:pipeline` moves the chart to `:complete`.

### chart exit (top-level final)  *(criticality: 8)*

When the chart enters a final that's a **direct child of the chart root** (a sibling of other top-level states), the chart itself terminates[^p4]:

```clojure
(statechart {}
  (state {:id :start} (transition {:event :go :target :end}))
  (final {:id :end}))                  ; top-level final
```

After entering `:end`:

- The configuration becomes empty: `#{}`.
- `running?` in working memory becomes `false`.
- Subsequent `t/run-events!` calls are silently dropped (no transitions can fire because nothing's active).

Distinct from "stuck" states (a regular state with no outgoing transitions): a stuck state stays in the configuration with `running?` still `true`; the chart could be re-driven via ancestor transitions. A chart that's exited can't be re-driven without a fresh env.

### internal event  *(criticality: 5)*

An event of `:type :internal` synthesized by the runtime, not from external `(t/run-events!)` calls. Examples in the library:

- `done.state.<id>` (this exercise)
- `done.invoke.<id>` (ex09)
- `error.execution.<id>` (when expressions raise exceptions; not covered in curriculum)

Internal events are queued and processed alongside external events, but they're generated by the chart's own machinery rather than your `t/run-events!` calls. The internal/external distinction matters mostly for `send` / `raise` semantics (advanced; not covered here).

### `done-data` (mentioned, not used)  *(criticality: 2)*

The library has a `done-data` element that can be declared *inside* (well — alongside; not as a child) a final to attach data to the auto-event:

```clojure
;; (Hypothetical; not in the exercise.)
(final {:id :done})
(done-data {} ...)
```

Per the library source, the `:data` of the auto-event is empty (`{}`) unless `done-data` is used. For the curriculum's exercise this doesn't come up; mentioned here for completeness when readers explore the library further.

### the "wait for all" pattern  *(criticality: 6)*

The pattern ex08 establishes:

> When multiple parallel tasks must all complete before the next step, declare each as a region with a final state, then handle `done.state.<parallel>` on the parallel's parent.

Common in:
- Page-load coordination (load A and B in parallel, render when both done)
- Multi-resource setup (open DB, open cache, open queue — all must succeed)
- Validation phases (run K independent checks, proceed when all pass)

The chart-level alternative — explicitly tracking completion in the data model with counters and after-each-event checks — is significantly more verbose and error-prone. The `final` + `done.state` mechanism is concise and matches SCXML's design intent.

### chart-exit cleanup (no built-in)  *(criticality: 3)*

The library doesn't provide an "on-exit-chart" hook. If you need cleanup when the chart exits (close DB connections, send a notification, etc.), you have two options:

1. **On-exit on the chart's top-level non-final states.** When the chart transitions to a top-level final, those non-final states exit first; their on-exit actions fire.
2. **Wrap the chart in a single top-level `state` whose only children are the chart's logical states + a final.** The wrapper state's on-exit fires when its final-child is entered.

Neither approach is in this exercise. Mentioned for awareness when designing production charts.

>
> - [^p1]: `(final {:id :x})` returns `{:id :x, :node-type :final}`
> - [^p4]: Top-level final entry → configuration empty + `running?` false
> - [^p5]: `done.state.<parallel>` fires only when ALL regions of the parallel are final
> - [^p6]: Finals can't have children; transitions inside throw `Illegal children of :final ...`
> - [^p7]: Auto-event shape: `{:type :internal :name :done.state.<parent-id> :sendid <final-id> :data {} ...}`
