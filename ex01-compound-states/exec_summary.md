# Executive Summary — ex01: Compound States & Configuration

The 5-minute refresher. Use this when you've completed ex01 once and want to re-anchor weeks or months later without re-reading `tutorial.md`.

---

## The one big idea

**A statechart is plain Clojure data, not a running process.** The `statechart` factory returns a map. The map doesn't *do* anything until you hand it to a runtime (a testing env, or in production a real env). The chart describes possible behavior; the runtime decides what happens.

Two consequences flow from this:

- The same chart definition can drive multiple independent envs (each with its own configuration and event queue).
- Re-evaluating `(def chart ...)` in the REPL doesn't update an existing env — that env captured the chart at creation time.

---

## What we built

A traffic light that cycles `:red → :green → :yellow → :red` on `:next` events.

```clojure
(def traffic-light
  (statechart {}
    (state {:id :red}    (transition {:event :next :target :green}))
    (state {:id :green}  (transition {:event :next :target :yellow}))
    (state {:id :yellow} (transition {:event :next :target :red}))))
```

Ten lines. Three atomic states under an implicit chart root (`:id :ROOT`, auto-generated by the factory). One event. A complete cycle.

---

## Mechanics in 30 seconds

- **The chart root is auto-named `:ROOT`.** You don't declare it; the `statechart` factory adds it for you. Treat the root like an implicit compound state — exactly one of its children is active at any time.
- **Document order picks the initial state** when no `:initial` is given. `:red` is the initial because it's declared first.
- **A transition is declared inside the source state.** It only needs to name the target (via `:target :id`). The "from" is implicit.
- **The configuration is a set** stored under `::sc/configuration` in working memory. After `(t/start! env)`, it's `#{:red}` — *just* the active atomic state.
- **`(t/in? env :x)` is `(contains? configuration :x)`** — a flat set-membership check.
- **Unknown events are silently dropped.** No error if you fire `:bogus`; the chart stays put.

---

## The four core API calls

| Call | Purpose |
| --- | --- |
| `(statechart {} ...)` | Build a chart definition (data) |
| `(t/new-testing-env {:statechart c ...} {})` | Build a runtime around the chart |
| `(t/start! env)` | Initialize the runtime; lands in initial state |
| `(t/run-events! env :event-name)` | Fire an event; chart settles into new config |
| `(t/in? env :state-id)` | Query the configuration set |

## Gotchas to remember

- **`(t/in? env :ROOT)` returns `false`**, even though `:ROOT` is the chart root's `:id`. The root is the *container*, not a state in the configuration set. *(Exception in ex03: `t/goto-configuration!` uses a different code path that DOES include `:ROOT`.)*
- **`(t/in? env :typo)` returns `false` silently** — no error for nonexistent states. `t/in?` is just `contains?`.
- **An empty `(statechart {})` builds and runs cleanly** with an empty configuration. Your exercise stub starts as this.
- **Forgetting `:id` on a state silently auto-assigns one** like `:state-12345`. Your `:target` references won't resolve.
- **`t/start!` called twice acts as a reset** — but the fresh-env-per-scenario pattern is the convention.

## Re-engage in 5 minutes

If you're returning to ex01 cold and need to verify your mental model:

```clojure
;; In a REPL:
(require '[exercises.ex01-compound-states :as ex01])
(require '[com.fulcrologic.statecharts.testing :as t])

(def env (t/new-testing-env {:statechart ex01/traffic-light
                              :mocking-options {:run-unmocked? true}} {}))
(t/start! env)
(t/in? env :red)          ; → true
(t/run-events! env :next)
(t/in? env :green)        ; → true
(t/run-events! env :next :next)
(t/in? env :red)          ; → true (cycled back)
```

If those four checks pass, ex01 is still solid. Move on to [ex02 (event matching)](../ex02-event-matching/) for hierarchical event names and the ancestor walk.
